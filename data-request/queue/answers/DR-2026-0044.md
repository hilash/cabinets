# DR-2026-0044 · Answer Packet

**Question:** Why did DAU drop on May 14? Is it a data pipeline issue or real?
**Requester:** Priya Mehta (Product Analytics)
**Priority:** P1 · Answered: 2026-05-14T12:10:00Z · SLA: met

---

## SQL Used

```sql
SELECT
  event_date,
  COUNT(DISTINCT user_id) AS dau,
  COUNT(DISTINCT CASE WHEN country_code IN ('DE','FR','NL','BE','AT') THEN user_id END) AS dau_eu
FROM fct_daily_active
WHERE event_date BETWEEN '2026-05-10' AND '2026-05-17'
  AND is_internal = false
GROUP BY 1
ORDER BY 1;
```

## Result

| Date | DAU (total) | DAU (EU) |
|---|---|---|
| 2026-05-10 (Sun) | 18,210 | 4,830 |
| 2026-05-11 (Mon) | 28,920 | 7,610 |
| 2026-05-12 (Tue) | 29,340 | 7,820 |
| 2026-05-13 (Wed) | 29,100 | 7,740 |
| **2026-05-14 (Thu)** | **24,100** | **2,280** |
| 2026-05-15 (Fri) | 28,650 | 7,590 |
| 2026-05-16 (Sat) | 17,980 | 4,710 |

## Interpretation

The DAU drop on May 14 is real and explained by a public holiday — **Ascension Day (EU bank holiday)** observed in Germany, France, Netherlands, Belgium, and Austria. EU DAU dropped from ~7,700 to ~2,280 (−70%), driving the overall DAU decline. Non-EU DAU was flat (~21,820 vs prior day ~21,360). No pipeline issue — the dbt model ran clean at 02:00 UTC, row count normal.

## Caveats

- Holiday calendar not currently encoded in the analytics layer — consider adding a `dim_holiday` reference table to flag expected low-DAU days automatically.
- EU user segmentation uses `country_code` from signup, not session IP — some EU users on VPN may be miscategorized.
